# session 模块

## 职责
agent 运行时核心。驱动「接收输入 → 调 LLM → 执行工具 → 持久化」的主循环，并实现两段式 delivery（steer/queue）、上下文压缩、会话恢复、title 生成、可中断（turn 边界 + mid-tool 硬中止）。

## 边界与非目标
- 不做 HTTP / 终端 IO（由 web/cli/tui 负责）。
- 不直接连数据库驱动——经 `Store` trait 抽象。
- 非目标：MCP / 权限确认（当前未实现）。
- skills 仅承接「可选的系统提示注入」：`SessionState.skill_prompt`（`Arc<Mutex<Option<String>>>`）是共享可变状态——TUI 持有 `Arc` 克隆并直接写入（绕过 cmd channel），`run_one_llm_call` 每 turn 经 `skill_prompt_cloned()` 读取最新值。每轮 `build_system` 把它作为 `## Active skill` 段追加到系统提示末尾（最高优先级）。skill 的发现/解析/选择 UI 不在本模块（见 core 的 `skill` 模块与 tui 的 `menu` 模块）。

## 关键抽象
- `SessionState`（`src/lib.rs`）：`id/messages/agent/model/config/client(Arc<dyn ChatStream>)/store(Option<Arc<dyn Store>>)/cancel(Option<CancellationToken>)/skill_prompt(Arc<Mutex<Option<String>>>)` + `working_dir/last_usage/persisted_count/session_created`。`cancel` 由调用方挂载（`with_cancel`）：web 经 `POST /interrupt`、tui 经双击 Esc 触发；同一 token 在 run_loop 顶部与 mid-tool select! 两处生效。`skill_prompt` 是共享可变状态：TUI 经 `Arc` 克隆直接写入（`set_skill`/`with_skill`），`run_one_llm_call` 每 turn 经 `skill_prompt_cloned()` 读取最新值，由 `build_system` 注入系统提示。此设计修复了 session 运行中激活 skill 不生效的 bug（旧设计经 cmd channel 发送 `SetSkill`，worker 阻塞在 `run_loop` 内无法及时处理）。
- `record(&mut self, msg)`（`src/lib.rs`）：push 到内存 + 若有 store 则持久化（best-effort，失败仅 warn）。runner 所有 message 入口都走它。
- `SessionEvent`（`src/runner.rs`，16 变体）：SSE/持久化的**单一真相源**三方法——`sse_kind()`（细粒度事件名字符串，如 `text_delta`/`reasoning_delta`/`subagent_child`/`plan_handoff`）、`sse_data()`（与 SSE 线格式一致的 JSON payload）、`coarse_kind()`（12 变体 `EventKind`，仅作 DB `type` 列与回退）。web（`SseEvt::from_session_event`/drain 持久化）与 TUI（`worker::persist_event`）都走这三方法，使 live 广播与 replay 的 kind/payload 完全一致；store 的 `sse_kind` 列保住 replay 细粒度（旧记录 `None` 时回退 coarse）。
- **事件缓冲 flusher**（`src/event_sink.rs`，导出 `EventSink`/`spawn_event_flusher`/`run_flusher`）：把 O(tokens) 的 token delta 落库降到 O(turn)。UI/SSE 实时投递路径（`on_event`）完全不动——每事件仍实时转发；只有磁盘路径走批量。`EventSink` 是可 clone 的 unbounded-channel push 句柄；`run_flusher` 只缓冲 `TextDelta`/`ReasoningDelta`（均 coarse-map `EventKind::TextDelta`），其余类型立即 flush；阈值 `DELTA_BATCH=512`/`DELTA_BYTES=8KiB`。无损契约：丢弃所有 sink clone + `await` flusher 句柄即保证剩余缓冲全落库（turn 边界/channel close 触发最终 flush）。崩溃最坏丢失当前 turn 少量未 flush delta，权威文本仍经 per-turn `messages` append 落地。调用方（TUI `worker`/web `handle`/headless）在 `on_event` 闭包内 `sink.push(&ev)`，run 结束后 `drop(sink)`+`flusher.await`。subagent（`runner.rs`）与 replay（`resume.rs`）子事件路径直接用 `run_flusher`+unbounded channel。
- `run_loop`（`src/runner.rs`）：drain 主循环。每轮顶部：① cancel 检查（turn 边界，web/tui 触发）② **claim_steers**（提升 pending steer，safe provider-turn boundary）③ 压缩判定 ④ 单次 LLM 调用 ⑤ 工具执行（**一批 tool_calls 并发**：`FuturesUnordered`，每个 future 开头 `yield_now().await` 保证全部被调度（否则快速响应退化为串行），`ToolStart` 按调用顺序预声明、`ToolEnd` 按完成顺序 emit，结果按原始 index 重排保证 Tool message 确定性；事件经共享 `Sink`——`Arc<Mutex<&mut (dyn FnMut+Send)>>` + `emit`——多发并发安全）⑥ 若无工具调用（idle）→ **claim_one_queued**（恰好一条，返回 `(seq,input)` 并 emit `QueueConsumed{seq}` 让前端即时收缩 pending 镜像），有则续跑，无则 Done。硬中止：`run_one_llm_call` 的流接收循环与 `execute_call` 的工具 `.await` 均包在 `tokio::select!` 中监听 `await_cancel(session)`；触发后前者回空 turn、后者回 `interrupted` 工具结果；工具批在 cancel 下仍 **drain 至完成**（不 break——否则丢弃在飞 subagent future 会漏发 `SubagentEnd`/`complete_subagent_task` 并留下无 result 的 tool_use），run 在下一轮顶部 cancel 检查收尾。bash 工具设 `kill_on_drop(true)`，select! 取消即丢弃 future → 杀子进程。**bash 进程经 `pre_exec(setsid)` 自成新会话/进程组（load-bearing：否则子进程仍共享控制终端、能直写 `/dev/tty`——sudo 提示/进度条/后台子进程的输出绕过 pipe、落到 alt 屏光标处即 TUI composer/input 区，正是『输出跑到输入区』bug 的根因；新会话令 `/dev/tty` 不可用、强制所有输出走被分捕的 pipe）**。timeout 时 `kill(-pgid, SIGKILL)` 杀整组（`setsid` 使 pid==pgid，故负 pid 一击尽杀整棵后代树，避免孤儿 grandchild；`kill_on_drop` 仅杀直接 bash 子，仅作 cancel/unwrap 回退，timeout 路径不依赖它）；随后回收部分输出（组杀后 pipe 写端关闭、drain task resolve，挂起前打印的内容是诊断关键线索，不再像旧版整段丢弃）。子 agent 复用父 token（`child.cancel = parent.cancel.clone()`）。doom-loop 守卫：`DOOM_THRESHOLD=3`，连续空 turn 达阈值后注入提示打破循环。**plan 模式 bash 写拦截**：`execute_call` 在 plan agent 调 bash 前调用 `bash_guard::classify`，若为写命令（文件重定向 `> file`/`>> log`/`&> out`、`tee <真实文件>`、变异/git写/包管理/就地编辑）则返回描述性错误给模型（"Blocked in plan mode"），不执行。判只读：fd-merge（`2>&1`/`1>&2`）与重定向到 `/dev/null`（含 `tee /dev/null`、裸 `tee`）——bash 工具已分捕 stdout/stderr 故无副作用；此判定在子 shell/花括号组内同样生效（`read_redirect_target` 在 `)`/`}`/`]`/`#` 处干净终止，不再误吞元字符）。plan agent 无 plan_exit 工具——计划以纯文本输出、turn 自然结束；用户 Shift+Tab 切 act 后 TUI 发 `SwitchAndStart` → worker 切 agent 后**先 `plan_handoff::handoff` 清空 transcript**（提取最后一条非空 assistant 文本作为「最终计划」，重置 `session.messages` 为单条合成 user 指令、发 `TranscriptReset` 让 UI `replay_into_chat` 重建为只含计划），再空 prompt 进 `run_loop`（系统提示变 act，act 以只含计划的干净 ctx 开始执行）。store 不动——append-only 保留完整原始记录供审计/resume（resume 仍 `load_messages` 重载全部历史，与压缩同理）。无计划文本时 `handoff` 返回 false、不改动（安全回退）。
- **subagent**（`src/runner.rs::run_subagent`）：`task` 工具触发子 agent 运行；子 SessionState 用独立 agent（explore 只读 / build 全工具，默认 explore）。**plan 模式 subagent guard**（`runner.rs:503`，先于 `resolve_agent`）：plan agent 仅可派发 `explore`，其它类型（含 `build`）以 "Unknown subagent_type" 拒绝（不提 build 存在）；act 模式不受限。plan prompt（`base_prompt_plan`）额外 `.replace()` 剥离 BASE_PROMPT 中的 build 广告子句，测试 `plan_prompt_strips_build_subagent_advertisement` 保护此不变式，事件经共享 `Sink` 转发到父级——`SubagentStart{id,kind,prompt,child_session_id}` / `SubagentEnd{id,ok,summary}`，外加 `SubagentChild{id, ev: Box<SessionEvent>}` 包装器——子回调把**所有**子事件（TextDelta/ReasoningDelta/ToolStart/ToolEnd/Done/Error）包进 SubagentChild 转发，而非仅透传 ToolStart/ToolEnd，供 TUI 路由到子 block 的独立 ChatView。`child_session_id` 供 TUI 加载子记录。子 agent 复用父的 client/store/cancel。一轮内多个 `task` 调用与其它工具调用**并发**执行（同上 `FuturesUnordered`）。**libsql 追踪**：若有 store，创建子 session 行 + `subagent_tasks` 记录（parent/child/prompt/result/status），子事件持久化到 `session_events`（fire-and-forget `tokio::spawn`），完成后更新 result + status。
- `compaction::should_compact`（`src/compaction.rs`）：双信号——token 估算（首轮即可触发，无需模型回传 usage）+ 模型回传 `last_usage.input_tokens`（**非** `total_tokens`，避免输出重的 turn 误触输入预算）；预算 = `min(context_threshold, context_limit - reserved)`，故 `reserved` 真正缩减可用窗口（不再是死字段）。摘要用 `small_model`。**摘要提示词为结构化模板**（`prompt::compaction_system_prompt` + `prompt::compaction_user_prompt`）：系统提示固定 anchored-context-summarization 角色；用户提示要求严格按 Objective / Important Details / Work State(Completed/Active/Blocked) / Next Move / Relevant Files 顺序输出 Markdown。`compact()` 先在 head 中查找既有合成 summary 消息（`[Conversation summary so far]` 前缀），剥前缀后经 `<previous-summary>` 块传入，summarizer **增量更新**而非重写。`compact()` 返回 `Option<String>`——`None` 表示 split==0（消息不足以压缩），caller 不 emit 任何事件；`Some(summary)` 时 runner 发 `TranscriptReset(new_msgs)` + `Compaction(summary)` 两个事件——TUI 收到 `TranscriptReset` 时清空 `chat.blocks` 并用 `replay_into_chat` 重建（显示与模型视图一致）。
- **reasoning_effort 出站透传 + 交错思考**（`src/runner.rs::run_one_llm_call` + `run_loop`）：主 LLM 调用把 `session.config.reasoning_effort`（`low|medium|high`）写进 `ChatRequest`，由 llm 的 `to_body` 发顶层字段（OpenAI 风格思考深度）；`compaction::summarize` / `resume::generate_title` 后台调用显式置 `None`。`run_one_llm_call` 返回 4-tuple `(text, reasoning, tool_calls, usage)`——reasoning 从 `LlmEvent::ReasoningDelta` 累积。**交错思考**（`interleaved_thinking` 默认 `Some(true)`）：`run_loop` 在工具调用 turn 且 reasoning 非空时，将 `ContentBlock::Reasoning { text }` 持久化到 assistant 消息（Text 之前），经 llm 的 `push_assistant` 序列化为 `reasoning_content` 在后续请求回传——DeepSeek-V4 强制要求此行为（否则 HTTP 400）。非工具 turn 不持久化（API 忽略，省 token）。运行时热重载：调用方（TUI `/model` 经 `UiCmd::ReloadConfig`）在 turn 边界调 `SessionState::apply_config_reload(cfg, new_client)` 原地替换 `client/model/config`。
- `resume::resume`（`src/resume.rs`，低层原语）：从 Store 重建 SessionState，model/agent 取自存储的 session meta（忠实原配置）。重建时两条 reconcile 路径：① 卡在 `Running` 的 subagent task（进程在 subagent 中途被打断）标记为 `Failed("(interrupted)")`；② 悬空的 `tool_use`（assistant 请求了工具但 tool_result 未落盘）合成 error `ToolResult`，避免 provider HTTP 400。这两条仅在裸 `resume` 直调时生效，是 `resume_and_replay` 之下的安全网。
- `resume::resume_and_replay`（`src/resume.rs`，高层入口；TUI/CLI/web 的 resume 都走它）：先遍历父 session 所有 `Running` subagent task（`list_subagent_tasks` 按 `seq` 升序），逐个 `resume(child)` + `run(child, "")`（**只此一层、不递归**——子 agent 无 `task` 工具，不可能派发孙任务），把子结果回填进父 transcript 的**单条** Tool message（mirrors `run_loop` 把一轮工具结果批进一条 tool_msg 的语义），并 `complete_subagent_task`。回填发生在裸 `resume` 之前，故 `resume` 跑到时无 `Running` task、task 的 `tool_use` 也已 answer，两条 reconcile 路径对该 turn 失活——既不重复标 Failed 也不合成 error。子续跑的 messages/events 落到子 session（与 `run_subagent` 同 pattern）。`generate_title` 用 small_model 异步生成标题。
- **plan→act handoff**（`src/plan_handoff.rs::handoff`）：plan→act 切换时把 transcript 压成单条合成 user 指令（最后一条非空 assistant 文本 = 最终计划 + 执行指令前缀）。与压缩同范式——纯内存 `session.messages` 重置 + caller 发 `TranscriptReset`；store append-only 不动。`final_plan_text` 取最新非空 assistant 消息；无则 `handoff` 返回 false 不改动。**边界落库**：`handoff()` 调 `SessionState::after_handoff(seq, plan)` 把移交边界（pre-handoff store message 数 + plan 显示文本）写入 `handoff_seq`/`handoff_plan`（由 TUI `SwitchAndStart` 经 `SessionPatch` 持久化）。`handoff_message(plan_display) -> Message`（合成 `Role::User`+`synthetic=true`）抽出复用。`resume()` 若 `meta.handoff_seq` 有值，则 trim `[0..handoff_seq]` 并 `insert(0, handoff_message(plan))` 重建焦点 transcript——用 `else if` 使**移交主导于压缩**（dominant transcript reset，二者互斥）。
- **能力门控工具集 + capability filter**（`src/tools/web_read.rs` / `web_fetch.rs` / `web_search.rs` / `computer_use.rs` + `src/runner.rs`）：`web_read`（always-on，markdown 协商 + `llms.txt` 爬取 + 可读正文抽取 + DDG 解析）、`web_fetch`/`web_search`（feature-gated `browser`，经 obscura，其 `!Send` V8/Rc future 包在 `spawn_blocking`+`LocalSet` 内以保 `Tool::execute` 仍 `Send`）、`computer_use`（always-on，驱动 core 的 `ComputerUseLoop`）。runner 在构建请求 schema 前以 `CapabilitiesConfig::tool_enabled(name)` 过滤——能力关时模型根本看不到该工具。这些可选工具收拢进 **`tools` umbrella 子 agent**（plan-visible，区别于 plan-hidden 的 `build`），由 act 经 `task` 调度。`ToolContext.proxy`（取自 `Config.network.proxy`）供工具执行时拿到代理配置。

## 主流程
CLI/HTTP 入口 → 建/恢复 SessionState（store 可选）→ `run(session, prompt, on_event)` → run_loop 循环到 Done / interrupt / doom-loop 守卫 → （CLI）异步 generate_title。

empty prompt = drain / continuation 模式：不 push 合成 user msg，直接进 run_loop（web drain 依赖 store 中已 admit 的 steer/queue 提供输入；TUI plan→act 手动切换经 `SwitchAndStart` 也走此路径——但**先经 `plan_handoff::handoff` 把 transcript 清为只含最终计划**，系统提示变 act，act 以干净 ctx 执行计划）。
